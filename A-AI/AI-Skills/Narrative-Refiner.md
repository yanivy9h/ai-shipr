# Skill: Narrative Refiner

## Purpose
Transform structured product thinking into clear, stakeholder-ready communication.
Preserve rigor. Remove noise. Adapt tone without losing substance.

## When to Use
- After structuring a decision or initiative — before presenting it
- When a document is technically correct but won't land with the audience
- When you need the same content to work for different stakeholder types
- Before writing a brief, memo, or async update

## Behavior Rules
- Require a specified stakeholder type — do not refine without a target audience
- Strip filler language, qualifiers without meaning, and buzzwords
- Preserve tradeoffs — never simplify to the point of dishonesty
- Adapt tone and emphasis, not the underlying facts
- Keep output shorter than the input where possible
- Never add hype, enthusiasm inflation, or reassurance that isn't earned
- Flag content that has been structurally weakened by vagueness — do not silently paper over it

## Input Expectations
Provide:
- Source content (initiative summary, decision brief, strategy section, etc.)
- Target stakeholder type:
  - **Engineering** — logic, constraints, technical tradeoffs, scope clarity
  - **Exec / CEO** — outcome, risk, strategic fit, decision needed
  - **Design** — user impact, experience tradeoffs, scope boundaries
  - **Sales / GTM** — customer impact, timing, competitive angle

Optional:
- Desired length (short / medium / full)
- Key message to preserve above all else

## Output Structure

### Refined Narrative
[Rewritten content adapted for target stakeholder]

### What Was Changed
- Removed: [...]
- Reframed: [...]
- Preserved: [...]

### Flags
- [Any content that was too vague to refine cleanly — identify specifically]
(Omit if none)
