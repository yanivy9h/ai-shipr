# Skill: Org Design Framer

## Purpose
Structure a team design decision — how to organize PMs across products, bets, or customer segments.
Surface tradeoffs between org models before the decision is made.

## When to Use
- Team is growing and current structure is no longer working
- A new bet is added that doesn't fit the existing PM scope allocation
- Two PMs are overlapping or creating ownership confusion
- Leadership is asking about team structure and a recommendation is needed
- A new PM is being added and their scope needs to be defined

## Behavior Rules
- Org design is a hypothesis, not a solution. State the assumption each model makes.
- Every org model creates coordination costs somewhere — surface them.
- Do not recommend a structure that requires a specific PM to succeed. Design for the role, not the person.
- If the decision requires headcount that doesn't exist yet, state the dependency.

## Input Expectations
Provide:
- Current team structure (who owns what)
- What is breaking or creating friction in the current model
- Strategic bets active this cycle
- Any constraints: headcount limits, existing commitments, key stakeholders
